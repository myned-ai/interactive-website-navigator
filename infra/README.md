# Infrastructure

Automated CI/CD pipeline using **GitHub Actions** to deploy the server to **Google Cloud Run** and the client to **Google Cloud Storage**.

## Architecture

```
GitHub (push to main)
  │
  ├─ server/ changed? ──► Build Docker image
  │                        Push to Artifact Registry
  │                        Deploy to Cloud Run (Vertex AI for Gemini)
  │
  └─ client/ changed? ──► Fetch Cloud Run URL
                           Build with VITE_WS_URL
                           Deploy to Cloud Storage (static site)
```

## Google Cloud Services Used

| Service             | Purpose                                      |
|---------------------|----------------------------------------------|
| Cloud Run           | Hosts the server (WebSocket + REST API)       |
| Vertex AI           | Gemini Live API access (no API key needed)    |
| Artifact Registry   | Docker image storage                          |
| Cloud Storage       | Static hosting for the client widget          |

## Prerequisites

- [Google Cloud CLI](https://cloud.google.com/sdk/docs/install) (`gcloud`)
- A GCP project with billing enabled
- A GitHub repository connected to the project

## Setup

### 1. Run the one-time setup script

> For production environments, we recommend replacing this script with an IaC tool such as Terraform or Pulumi for state tracking, drift detection, and reproducible teardown.

**macOS / Linux:**
```bash
./infra/setup.sh
```

**Windows (PowerShell):**
```powershell
.\infra\setup.ps1
```

This script:
- Enables required GCP APIs (Cloud Run, Artifact Registry, Vertex AI, Cloud Storage)
- Creates an Artifact Registry repository for Docker images
- Creates a Cloud Storage bucket with static website hosting
- Creates a service account for GitHub Actions with minimal IAM permissions
- Grants the Cloud Run default service account access to Vertex AI (Gemini)
- Generates a service account key and prints the GitHub Secrets to configure

### 2. Add GitHub repository secrets

After running `setup.sh`, add these secrets in **GitHub → Settings → Secrets and variables → Actions**:

| Secret               | Value                                            |
|----------------------|--------------------------------------------------|
| `GCP_PROJECT_ID`     | Your GCP project ID                               |
| `GCP_SA_KEY`         | The full JSON service account key from `setup.sh` |
| `GCP_CLIENT_BUCKET`  | The bucket name printed by `setup.sh`             |
| `GCP_AUTH_SECRET_KEY` | The auth secret key generated by `setup.sh`       |

After adding the secrets, delete the local key file generated by the setup script:

**macOS / Linux:**
```bash
rm /tmp/github-actions-key.json
```

**Windows (PowerShell):**
```powershell
Remove-Item $env:TEMP\github-actions-key.json
```

### 3. Push to main

```bash
git push origin main
```

The workflow will automatically detect which folders changed and deploy only what's needed.

## How Deployment Works

The GitHub Actions workflow (`.github/workflows/deploy.yml`) runs on every push to `main`:

1. **Detect changes** — compares `HEAD~1` to determine if `server/` or `client/` files changed
2. **Deploy server** (if changed) — builds the Docker image, pushes to Artifact Registry, deploys to Cloud Run with Vertex AI environment variables
3. **Deploy client** (if changed) — fetches the Cloud Run URL, builds the client with `VITE_WS_URL` pointing to the server, uploads to Cloud Storage

If both folders changed, the server deploys first so the client can resolve the server URL.

## Manual Deployment

### Server

```bash
# Build and push manually
gcloud builds submit server \
  --tag europe-west1-docker.pkg.dev/PROJECT_ID/interactive-website-navigator/interactive-website-navigator:latest

# Deploy to Cloud Run
gcloud run deploy interactive-website-navigator \
  --image europe-west1-docker.pkg.dev/PROJECT_ID/interactive-website-navigator/interactive-website-navigator:latest \
  --region europe-west1 \
  --allow-unauthenticated
```

### Client

**macOS / Linux:**
```bash
cd client
VITE_WS_URL=wss://YOUR_CLOUD_RUN_URL/ws npm run build
gsutil -m rsync -r -d dist gs://YOUR_BUCKET_NAME
```

**Windows (PowerShell):**
```powershell
cd client
$env:VITE_WS_URL="wss://YOUR_CLOUD_RUN_URL/ws"; npm run build
gsutil -m rsync -r -d dist gs://YOUR_BUCKET_NAME
```

## Local Development

For local development, the server uses an API key instead of Vertex AI:

```bash
# server/.env
GEMINI_USE_VERTEX=false
GEMINI_API_KEY=your-api-key-here
```

The client defaults to `ws://localhost:8080/ws` when `VITE_WS_URL` is not set.

## Teardown

To remove all GCP resources created by the setup script:

**macOS / Linux:**
```bash
./infra/teardown.sh
```

**Windows (PowerShell):**
```powershell
.\infra\teardown.ps1
```

This deletes the Cloud Run service, Artifact Registry repository, Cloud Storage bucket, service account, and IAM bindings. After running, also remove the GitHub repository secrets (`GCP_PROJECT_ID`, `GCP_SA_KEY`, `GCP_CLIENT_BUCKET`, `GCP_AUTH_SECRET_KEY`).
